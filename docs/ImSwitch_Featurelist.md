# Newswitch feature inventory — what ImSwitch ships today

_Source: ImSwitch `master`, current branch `fix/streaminglatency`. Generated 2026-05-29._

This is a deep audit of every microscope-relevant feature that lives in this
repo today, grouped by area, with backend and frontend listed side by side.
It is the input for the newswitch port — for each feature we mark a
**destination**: stays in **newswitch**, migrates to **Arkitekt** (distributed
orchestration), migrates to **Optikit** (optical layout, firmware, module
library), or is **dropped** as redundant / out-of-scope.

Companion docs:

- [ImSwitch_Functionality_Overview.md](ImSwitch_Functionality_Overview.md) — older, shallower bullet list (Feb 2026)
- [MIGRATION.md](MIGRATION.md) — newswitch port tracker (phase/PR table)
- [STREAMING_REVIEW.md](STREAMING_REVIEW.md), [STREAMING_4B_NOTES.md](STREAMING_4B_NOTES.md), [ReviewCameraStreamingIssues.md](ReviewCameraStreamingIssues.md) — streaming subsystem deep-dives
- `~/Downloads/optikit_architecture.svg` — target ecosystem diagram

## 0. How this doc is organised

Sections **1** through **12** are the feature groups, ~70 controllers and
~50 frontend components mapped onto ~20 areas. Each entry lists:

- **Backend** — controllers in [imswitch/imcontrol/controller/controllers](imswitch/imcontrol/controller/controllers) and managers in [imswitch/imcontrol/model/managers](imswitch/imcontrol/model/managers).
- **Frontend** — React components in [frontend/src/components](frontend/src/components) and [frontend/src/axon](frontend/src/axon), Redux slice in [frontend/src/state/slices](frontend/src/state/slices), axios wrappers in [frontend/src/backendapi](frontend/src/backendapi).
- **Endpoints (n)** — number of `apiXxxControllerYyy.js` files = number of `@APIExport`-decorated REST methods on the controller.
- **Capabilities** — what the feature actually does.
- **Destination** — `→ newswitch`, `→ Arkitekt`, `→ Optikit`, or `→ drop`.

Sections **13** through **17** are cross-cutting analyses: what moves where,
what's missing, what's redundant, what's plain dead, and a recommended port
order.

Headline counts (all in current repo):

- **70 controllers** (`imcontrol/controller/controllers/*.py`)
- **30 managers** (`imcontrol/model/managers/*.py`) + 30+ hardware-specific manager subclasses
- **260 axios endpoint wrappers** (`frontend/src/backendapi/api*.js`)
- **~75 React components** (`frontend/src/components/` + `frontend/src/axon/`)
- **~50 Redux slices** (`frontend/src/state/slices/`)
- **~20 vendor camera/laser/stage backends**

## 1. Shared building blocks (cross-cutting infrastructure)

These aren't features per se but every feature touches them, so the port plan
hinges on them.

### Backend framework

- **`ImConWidgetController` / `LiveUpdatedController`** — base classes for every controller. Provide `_master` (manager registry), `_widget` (Qt widget — legacy), `_commChannel` (Signal bus), `_setupInfo` (parsed setup JSON). Every controller `__init__` looks like a service constructor.
- **`@APIExport()` decorator** ([imswitch/imcommon/model](imswitch/imcommon/model)) — turns a method into a FastAPI POST/GET. Optional `requestType="GET"|"POST"` and `runOnUIThread=True/False`. This is the only mechanism for the entire REST surface.
- **`Signal`/`Worker`/`Thread`/`Timer`** ([imswitch/imcommon/framework/noqt.py](imswitch/imcommon/framework/noqt.py)) — Qt-free shim that emulates `QtCore.Signal` so controllers can `sigXxx.emit(...)` from any thread. Signals are bridged to Socket.IO emits in headless mode.
- **`MultiManager`** — generic singleton container for hardware. `_master.detectorsManager`, `_master.lasersManager`, `_master.positionersManager`, etc. — every manager is keyed by the device name from the setup file.
- **Setup files** — `imcontrol_setups/*.json`. Define detectors, lasers, positioners, objectives, focus-lock geometry, wellplate geometry, available widgets. This is the de-facto microscope description language today.
- **`headless` mode** ([main.py](main.py), [main2.py](main2.py)) — runs `applaunch.py` without instantiating Qt widgets. The whole React frontend assumes this mode.

### Frontend framework

- **CRA + Material-UI v5 + Redux Toolkit + react-redux** — ageing CRA stack, not Vite.
- **`WebSocketContext`** ([frontend/src/context/WebSocketContext.js](frontend/src/context/WebSocketContext.js)) — Socket.IO client provider. Every component subscribes to specific event names that map 1:1 to `sigXxx` Signals in Python.
- **`middleware/WebSocketHandler.js`** — hand-routes incoming socket events to Redux actions. Currently being patched on `fix/streaminglatency` for stream backpressure.
- **`backendapi/api*.js`** — 260 hand-written axios wrappers, one per `@APIExport` method. Generated by hand, drift constantly.
- **`state/slices/*.js`** — ~50 Redux slices, one per logical feature. Every slice has the same `setX/setY` boilerplate; no type contract with Python.
- **`hooks/useBackendControllerCapabilities.js`** — runtime probe of `GET /controllers/list` so the UI can hide features unavailable in the current setup. The only "capability registry" today.
- **`stream/uc2f.js`** — UC2 binary frame protocol decoder (Zstd-compressed raw frames over Socket.IO).

### Newswitch destination for the shared layer

| Block                                   | Destination                                                         |
| --------------------------------------- | ------------------------------------------------------------------- |
| `@APIExport` decorator                  | **drop** — replaced by `@register` (rekuest-next)                   |
| `Signal` → Socket.IO bridge             | **drop** — replaced by `@state` field mutation → `STATE_UPDATE`     |
| Setup JSON                              | **→ Optikit** — module YAML index + universal coordinate origin     |
| `MultiManager` pattern                  | **→ newswitch** — recast as Protocol + injected state class         |
| Redux slices, axios wrappers, WS bridge | **drop** — generated Zod hooks + scoped Zustand replace all of them |
| CRA build                               | **drop** — newswitch is Vite                                        |

### Capabilities

- Three coexisting transports: **MJPEG** (HTTP multipart), **Socket.IO JPEG** (per-frame base64), **Binary Socket.IO** (Zstd/LZ4-compressed raw frames). The latter is the modern path.
- Per-detector stream parameters: target FPS, quality, downscale, crop, binning, color/mono.
- WebGL viewer + OpenLayers viewer + native `<img>` MJPEG viewer; user selects.
- WebRTC H.264 path via [WebRTCController.py](imswitch/imcontrol/controller/controllers/WebRTCController.py) (separate aiortc server) — low latency, browser-decoded.
- Histogram overlay, grid + crosshair, picture-in-picture, frame stats (FPS, KB/s, dropped frames).
- 3D viewer for lightsheet stacks; Vizarr browser viewer for OME-Zarr.

### Destination

- Streaming **architecture** (StreamWorker with codec adapters, per-detector parameters, binary-frame protocol): **→ newswitch** Phase 5 — port the design, not the code. The dataclass-based `StreamParams` contract is a good template for a Zod schema.
- WebRTC: keep as an *optional transport* in newswitch for sub-frame latency; do not let it be a parallel control plane.
- MJPEG path: **→ drop** in favour of binary frames (kept today only as fallback for low-spec clients).
- Histogram + LUT + overlays: **→ newswitch** as client-side concerns over the binary frame buffer.
- Vizarr / 3D viewers: **→ newswitch** UI; the Zarr files themselves are served by the storage node (Arkitekt-registered).

## 24. What is missing for a complete-microscope baseline

(Judged against what a working multi-modal microscope needs end-to-end, not
against what ImSwitch happens to have.)

- **First-class Z-stack primitive** — today buried inside `ExperimentController`; should be `run_zstack(positions, channels, z_range, z_step)` (already in MIGRATION.md Phase 3).
- **First-class channel sequencing** — `CompositeController` is partial; what is missing is a typed "channel preset" (illumination source + intensity + exposure + gain + filter) that every acquisition routine consumes.
- **Per-channel histogram + LUT** — `HistogrammController` has exactly 1 endpoint; the frontend lacks a real histogram + LUT editor for multichannel data.
- **Per-channel acquisition presets with recall** — no equivalent to MicroManager's "preset groups".
- **Multi-detector simultaneous acquisition** — manager supports multiple detectors, no routine starts them in sync.
- **Hardware-sync dual-camera trigger** — the `TriggerAcquisitionController` is single-camera.
- **Live cross-correlation autoalign** — would supersede the camera-based `FOVLock`.
- **Universal coordinate system** — Optikit diagram calls this out as zone B; ImSwitch only has stage-relative coordinates, not a build-absolute origin. *This is the single biggest architectural gap.*
- **Acquisition queue / job manager** — start later, run while microscope reboots, retry on failure. Today every controller runs synchronously.
- **Cancellation that stops camera + lasers + stage cleanly** — today every controller invents its own stop logic.
- **Provenance log** — metadata is embedded in OME files, but there is no audit trail of "who did what when".
- **Per-user / per-session permissions** — relevant for shared microscopes.
- **Hardware-health telemetry** — drive uptime, motor steps, laser hours, calibration drift. The Optikit diagram zone F explicitly calls for this for after-sales; ImSwitch has none.
- **Image annotations from analysis drawn over live view** — only manual ROIs today.
- **OME-Zarr live append** — recording into Zarr today is by chunked write at end; live-append + tile-mosaic-into-Zarr should be the default.
- **Native digital-twin sync back to Optikit** — the configurator should see live focus, stage position, current channel; today this is implied but not implemented.

## 26. Recommended newswitch port order

Combining MIGRATION.md's phase table with this audit and the Optikit/Arkitekt
end-state:

1. **Phase 0 (audit)** — this document closes it. Sign off on the destination matrix in §23.
2. **Phase 1 (protocols + virtual + UC2)** — extend MIGRATION.md table with `FilterBank`, `Galvo`, `Autofocus`, `FocusLock`, `LEDMatrix`. Virtual first, UC2 second.
3. **Phase 2 (core actions)** — done for stage, in-progress for illumination/detector/objective.
4. **Phase 3 (workflow routines)** — port in this order: `move_to_well` (tiny, proves pattern) → `run_zstack` → `run_timelapse` → `run_tile_scan` → `run_autofocus` → `run_focus_lock_calibration` → `run_wellplate_experiment` → `compute_focus_map` → split `ExperimentController` last.
5. **Phase 4 (UI parity)** — one React component per routine; reuse axon/experiment-designer dimension editors directly (they are clean of Redux).
6. **Phase 5 (streaming)** — port the `StreamWorker` design (codec adapters, per-detector params, binary frame protocol). Default = binary Zstd; WebRTC optional; MJPEG dropped.
7. **Phase 6 (setup-file)** — replace `imcontrol_setups/*.json` with Optikit YAML index + universal coordinate origin. This is the *hard* phase, gated on Optikit module library being usable.
8. **Phase 7 (cutover)** — see MIGRATION.md gate.

Two transverse efforts that should run alongside, not after:

- **Calibration → Optikit write-back**: as each calibration routine ports, wire it to write its result into the Optikit module DB so the new "single source of truth" claim holds from day one.
- **Provenance + telemetry node**: stand up an Arkitekt-registered storage + provenance node early so every dataset has lineage from the first port.
